o manage 200 missions with a live financial heartbeat, the Zustand store must be high-performance. We can't afford to re-render the entire dashboard every time a bot makes $0.01.

We will use a "Slices" pattern to separate the 200 static Mission definitions from the high-frequency Live P&L and Log data.

1. The High-Performance Store (useSovereignStore.js)
This store handles the mission registry, the active running state, and the cumulative profit tracking.

JavaScript
import { create } from 'zustand';
import { subscribeWithSelector } from 'zustand/middleware';

export const useSovereignStore = create(subscribeWithSelector((set, get) => ({
  // 1. STATIC REGISTRY (The 200 Missions)
  registry: [], 
  setRegistry: (data) => set({ registry: data }),

  // 2. LIVE STATE (Running Missions & Metrics)
  activeMissions: {}, // Keyed by Mission ID for O(1) access
  globalMetrics: {
    totalProfit: 0,
    activeAgents: 0,
    gasSpent: 0,
    lastUpdate: Date.now()
  },

  // 3. ACTIONS
  deployMission: (missionId) => {
    const mission = get().registry.find(m => m.id === missionId);
    if (!mission) return;

    set((state) => ({
      activeMissions: {
        ...state.activeMissions,
        [missionId]: { 
          ...mission, 
          status: 'INITIALIZING', 
          pnl: 0, 
          logs: [], 
          startTime: Date.now() 
        }
      },
      globalMetrics: { ...state.globalMetrics, activeAgents: state.globalMetrics.activeAgents + 1 }
    }));
  },

  // 4. HIGH-FREQUENCY UPDATES (Called via WebSocket)
  updateMissionMetrics: (missionId, deltaPnL, newLog) => {
    set((state) => {
      const mission = state.activeMissions[missionId];
      if (!mission) return state;

      const updatedMission = {
        ...mission,
        pnl: mission.pnl + deltaPnL,
        logs: [newLog, ...mission.logs].slice(0, 50), // Keep last 50 logs
        status: 'EXECUTING'
      };

      return {
        activeMissions: { ...state.activeMissions, [missionId]: updatedMission },
        globalMetrics: {
          ...state.globalMetrics,
          totalProfit: state.globalMetrics.totalProfit + deltaPnL,
          lastUpdate: Date.now()
        }
      };
    });
  },

ðŸ›ï¸ 2. The Deployment Pipeline (The Bridge)
When you hit "HOT_RELOAD_LOGIC", the React UI doesn't just save a file. It triggers a system-wide event:

Stage: The new code is sent to Dept 01 via the API.

Verify: Dept 01 runs a "Shadow Container"â€”it executes the new logic on real data but doesn't sign any transactions (Read-only).

Compare: If the "Shadow" output matches the "Live" output (or improves it), the OS performs a Blue-Green Deployment.

Cutover: The old worker is gracefully shut down after finishing its current Kafka message, and the new worker takes the next offset.
